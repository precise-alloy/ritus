# Execute Task

**Core principle:** Implement exactly what the task file says — no more, no less. If a step is unclear, stop and
report BLOCKED instead of guessing.

## When to use

When given a task file path to implement. This skill defines how to execute task files generated by `ticket-review`
or `debug` skills.

After implementation, dispatch a fresh subagent (model: haiku, effort: medium) to run the `verify-task`
skill with the task file path and your diff. Do not self-verify — the reviewer must have a clean context.

## Your role

You are a **code executor**. You receive task files and implement them.

- Implement STEPS exactly as written
- Do not redesign, re-triage, or re-plan
- If a STEP is unclear: stop and report `BLOCKED: <reason>`. Do not guess.

## Steps

1. Read task file. Two formats exist:
   - **SIMPLE** (3-section): TASK + DONE WHEN + VERIFY only. Skip steps 2, 7, 12 below.
   - **STANDARD/EPIC** (full): TASK, PATTERN, CONTEXT, GOAL, STEPS, DONE WHEN, VERIFY, DOC UPDATE, COMMIT.
2. Read PRIOR CONTEXT if present.
3. Read CONTEXT files listed in the task. Grep for more if needed — no full repo scan.
4. Read `docs/tasks/{branch-slug}/exploration.md` if it exists — prior findings from other subagents on this branch.
5. **If task is a bug fix:** grep `docs/LESSONS.md` and `docs/DECISIONS.md` for affected module — load matching entries
   only.
6. Load applicable standard skills — **blocking step, not optional:**

   <!-- Keep in sync with start-ritus/SKILL.md and verify-task/SKILL.md -->

   | Task touches | Load skill |
   |---|---|
   | Any code change | `code-conventions` |
   | New service / endpoint / worker / bug fix | `testing-policy` |
   | New business logic, new API endpoint, or bug fix | `tdd` |
   | Auth / billing / migration / tenant isolation / infra config / shared contracts | `security` |
   | Any STANDARD or EPIC task | `definition-of-done` |

7. If PATTERN set: read `docs/ARCHITECTURE.md` for the relevant checklist.
8. Implement STEPS.
9. Run smoke test if in DONE WHEN.
10. Run unit tests if in DONE WHEN.
11. Update docs per DOC UPDATE.
12. Append new codebase findings to `docs/tasks/{branch-slug}/exploration.md` — use flagged format from the
    exploration template. Create the file if it doesn't exist.
13. Report using the output format below.

## Output format

```text
Files changed:
- path — summary

Docs updated:
- path — what changed   (or: none required)

Commit message:
type(scope): subject

- what changed and why
- key invariant enforced (if any)

Breaking: none | <what breaks>
Migration: none | <migration needed>
```

Commit types: `feat` | `fix` | `refactor` | `test` | `docs` | `chore`

## Context hygiene

Keep your working context clean to preserve performance and reduce cost:

- **Grep results**: summarize to key findings (file paths + line numbers). Do not dump raw output.
- **File reads**: read only the relevant section, not the entire file, when a section suffices.
- **Test output**: capture pass/fail summary and failing test names. Drop full logs unless debugging a failure.
- **Retries**: after a retry succeeds, drop the failed attempt's output. Keep only the working result.
- **Duplicate reads**: do not re-read a file you already have in context.

## Hard rules

1. Minimal change — implement only what STEPS specify. No unsolicited refactors.
2. Grep before edit — confirm file paths exist before touching any file.
3. No hallucinated features — if a STEP references something that doesn't exist, report BLOCKED.
4. Stop on errors — compile fail, test fail, 4xx/5xx → stop immediately and report.
5. Done = every DONE WHEN checkbox verified. Report what you did — verification is `verify-task`'s job.
6. Update docs per DOC UPDATE section before reporting done.
7. One final report — no intermediate dumps.
8. Grep before claiming — cite `file:line` when asserting any fact about existing code.
9. Finish the work — before ending your turn, check your last output. If it describes work you haven't done yet
   ("I'll now...", "next I should..."), do that work with tool calls instead of stopping.
