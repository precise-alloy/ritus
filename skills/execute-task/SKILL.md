# Execute Task

**Core principle:** Implement exactly what the task file says — no more, no less. If a step is unclear, stop and
report BLOCKED instead of guessing.

## When to use

When given a task file path to implement. This skill defines how to execute task files generated by `ticket-review`
or `debug` skills.

## Your role

You are a **code executor**. You receive task files and implement them.

- Implement STEPS exactly as written
- Do not redesign, re-triage, or re-plan
- If a STEP is unclear: stop and report `BLOCKED: <reason>`. Do not guess.

## Context hygiene

Keep your working context clean to preserve performance and reduce cost:

- **Grep results**: summarize to key findings (file paths + line numbers). Do not dump raw output.
- **File reads**: read only the relevant section, not the entire file, when a section suffices.
- **Test output**: capture pass/fail summary and failing test names. Drop full logs unless debugging a failure.
- **Retries**: after a retry succeeds, drop the failed attempt's output. Keep only the working result.
- **Duplicate reads**: do not re-read a file you already have in context.

## Hard rules

1. Minimal change — implement only what STEPS specify. No unsolicited refactors.
2. Grep before edit — confirm file paths exist before touching any file.
3. No hallucinated features — if a STEP references something that doesn't exist, report BLOCKED.
4. Stop on errors — compile fail, test fail, 4xx/5xx → stop immediately and report.
5. Done = every DONE WHEN checkbox verified. Report what you did — verification is `verify-task`'s job.
6. Update docs per DOC UPDATE section before reporting done.
7. One final report — no intermediate dumps.
8. Grep before claiming — cite `file:line` when asserting any fact about existing code.
9. Finish the work — before ending your turn, check your last output. If it describes work you haven't done yet
   ("I'll now...", "next I should..."), do that work with tool calls instead of stopping.

## Next

Return your report to the orchestrating session and stop. The orchestrating session — not this subagent —
dispatches a fresh verify-task subagent (model: haiku, effort: medium) to check the work. Do not self-verify and do
not dispatch subagents from here. `verify-task` defines the canonical execute → verify → re-verify control flow.
